# Architecture — L9 Node Repo Template

## System boundary

The L9 Node Repo Template is a static golden substrate for generated L9 node repos. The boundary is: spec in, generated repo out.

## Ownership map

| Concern | Owner | Template rule |
|---|---|---|
| Repo layout | repo template | owned directly |
| Contracts | repo template | reference contracts sourced from `nodespec.yaml` |
| Cursor rule templates | repo template | rendered and drift-checked |
| Governance and CI | repo template | owned directly |
| Observability scaffold | repo template | owned directly |
| `nodespec.yaml` parsing | l9-codegen-engine | external ownership |
| Handler generation | l9-codegen-engine | external ownership |
| Codegen determinism | l9-codegen-engine | external ownership |
| Agent doctrine and evals | l9-harness-library | external ownership |

## Transport bridge

`src/l9_service/enginehandlers.py` is the sole SDK transport bridge. It imports `TransportPacket` directly and must be generated by `l9-codegen-engine` from `nodespec.yaml`. The template validates this file as a generated-only target and must not contain handler generation logic.

## Runtime app

`src/l9_service/main.py` provides a FastAPI health/readiness shell. It is template infrastructure, not proof of concrete node behavior.

## Domain package

`src/l9_service/domain/` contains pure domain types and logic. It must not import SDK transport types. Generated handlers may call domain code, but domain code must not own routing or transport.

## Validation gates

- `tools/verify_contracts.py` verifies required contract files and references.
- `tools/validate_generated_boundaries.py` verifies generated-file markers and transport boundary.
- `tools/audit_engine.py` performs static L9 rule checks.
- `tools/review/analyzers/template_compliance.py` verifies template compliance.
- Cursor rules are rendered from templates and checked for drift.
